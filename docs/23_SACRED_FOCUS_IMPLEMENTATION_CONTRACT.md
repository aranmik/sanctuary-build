> 1. 성심 집중은 치유량을 늘리는 손이 아니라, 마나의 '시간'을 앞당겨 **수습의 예산을 미리 벌어두는 손**이다.

## 2. 코어 무접촉 위험 분석 (실증 프로브 P1~P6b 첨부)

**R1 — 클로저 봉인 (이 판의 물리 법칙)**: payMana/useSkill은 `createGame()` 클로저 내부 함수이고 반환 객체에 payMana가 없다(977). 따라서 **함수 가로채기(몽키패치)는 원천 불가** ⛔. "코어 무접촉 구현"이 성립하는 유일한 접촉면은 반환된 상태 `S`뿐이다.

**R2 — 허용 경로는 정확히 2종**: ⓐ **지불 후 정확한 복원(환급)** — `useSkill` 동기 호출 전후로 `mana/manaMin/oomT` 3필드를 스냅샷·복원. 단일 스레드 동기 구간이라 일시 하락은 관측 불가(P2: 부동소수까지 정확 일치 실증). ⓑ **상태 주입** — 어댑터 루프에서 마나 가산(P6a/6b: 이득 55.2 정확·캡 준수·통계 무접촉 실증). 둘 다 renew/seed의 직접 차감·GCD 세팅이라는 **기성 선례 위에 서 있다.**

**R3 — 사전 게이트의 의미 한계**: 721행이 지불 '전'에 잔액을 검사하므로 환급형 무료는 **"잔액≥비용일 때만 발동 후 환급"**이다. 진짜 0마나 시전은 불가(P5: 잔액 10 주입 시 deny '마나가 부족합니다' 실측). oom 문턱(40)과 비용대(35~100)가 겹쳐, **포커스가 가장 빛나야 할 바닥 순간에 침묵할 수 있다.** 단 renew/seed는 게이트도 어댑터 소유(1340/1301)라 진짜 무료(0마나 시전)까지 가능 — 두 '무료'의 의미가 갈라진다(§5-③).

**R4 — 시전형 환급 = 루프 감시자**: big/pray의 지불은 step 내부(870)에서 일어나 동기 래퍼가 못 잡는다. step 전 `p.cast` 스냅샷 → step 후 비교로 **결정론적 구현 자체는 가능**하나, 코어 내부 순서(868 회복→870 resolve)에 결합돼 코어 개정 시 **침묵 파손** 위험. 반대급부: 무산(776)·취소(712)는 지불이 아예 없어 **포커스 보존이 자연 성립**(P3·P4).

**R5 — 통계 정합**: 환급 시 manaMin/oomT 오염은 스냅샷 복원으로 소거됨(P2). 주입(C-1)은 payMana를 안 타므로 애초에 통계 무접촉(P6a).

**R6 — UI 지뢰 2곳**: `skCostTxt`(1367)·베일 루프(1630)가 `CFG.skills[k]` 무방비 접근 → 'focus' 키 분기 누락 시 **렌더 TypeError**. `ldName` 리졸버도 분기 필요.

**R7 — 스모크 불변성**: 무입력 런은 스킬 0회 → 어떤 안이든 버튼·상태 추가만으로는 51.4에 산술 영향 0. 단 A안의 루프 감시자는 step 전후에 상주하므로 **'무입력 시 no-op'을 유닛으로 못 박아야** 한다.

**R8 — 기회비용**: 풀 8→9, 장착 6 고정. 성심 집중을 들면 기존 6종 중 하나를 **내려놓아야** 한다 — EP07 "이거 대신 이거…" 감정선과 정합(손맛 자산 🔷).

**R9 — 명칭**: 카탈로그 1259에 3층 명명 기등록(display 성심 집중 / origin 내면의 집중력 / impl defined). Store Naming Audit 부담 최소.
