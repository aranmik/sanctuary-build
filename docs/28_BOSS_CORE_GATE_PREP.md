## 4. 개방안 3안 비교 (🔷 — 결정은 Boss Core Gate Decision 01에서)

### A안 — 최소 분기형 (구조 추천)
- **모양**: 코어 내 `BOSS_DEFS={morgas:{boss:<현 CFG.boss>,script:<현 SCRIPT>}, crusher:{…}, abyss:{…}}` + `createGame(bossId='morgas')`에서 `B/SC` 룩업 + 반환에 `script:SC` + 리터럴 7곳 `B.name` 보간. 어댑터는 전선 3줄·예보 재배선 2줄·HTML 동적화 2줄·셸 해금.
- **코어 편집 표면적(추정)**: 십수 줄 — §2 표의 ①③④⑤가 전부.
- **장점**: diff 최소·리뷰 가능 · 무인자 폴백으로 스모크 호출 불변 · **파쇄자는 기존 어휘 6종만으로 구현 가능**(스루풋형 — 신규 사고 타입 0) → 게이트 첫 검증이 저위험 · `BOSS_DEFS` 명명/형태를 팩 호환으로 두면 **B안의 부분집합**(승격 경로 확보).
- **위험**: 심연은 어휘 확장 +1~2 필요(저강도 지속 광역·마나 압박 — 기존 judge/brand의 재조합으로 부족 시 fireScript 스위치 확장=코어) → **파쇄자와 심연의 게이트 청구서가 다름**(순차 권장 근거). 코어 바이트 변경 자체는 불가피(재-baseline).
- **검증 난이도**: 중 — 기존 51.4 회귀 + 보스별 신규 사망열 박제.

### B안 — 보스 팩 구조형
- **모양**: `BOSS_PACKS`에 boss·script·패턴명(NM)·리포트 태그·예보 매핑까지 동봉, 코어가 팩을 소비(로그 문장 템플릿화 포함).
- **장점**: 3번째 이후 보스가 "데이터 추가"에 수렴 · 어댑터-코어 어휘 결합(§1 관찰) 해소 · 장기 5~7종 지향.
- **위험**: 코어 개정 폭 큼(fireScript 데이터 구동화 리팩터 — 사고 로직 완전 데이터화는 환상이라 경계 설정이 어려움) · 전 보스 리그레션으로 검증 폭발 · **현재 수요 2종에 과설계** · 재-baseline 후 diff 가독성 저하.
- **검증 난이도**: 상.

### C안 — 완전 데이터화 보류형 (시점 추천)
- **모양**: 게이트를 지금 열지 않는다. 파쇄자/심연은 셸·출정 경고(Matrix→Sortie Warning 01)까지만 강화, 실전투는 **노트북 이관 후 새 기준선에서**.
- **장점**: 기준선 3숫자 무손상(§3 시점 리스크 해소) · 이관 절차(100 §4)가 단순한 채 유지 · 27 §7 순서·100 §6 이관 후 첫 작업 목록과 정합.
- **위험/손실**: 데모 선언 불가 상태(27 §2 ⚠️) 지속 · 세 질문 검증 지연 — 단 이는 "지연"이지 "포기"가 아님.

### 렌 추천 (합의 요청)
**구조는 A안, 시점은 C안** — 즉 "이관 후 첫 코드 카드에서 A안으로 게이트를 연다."
근거: ① 27 §9·100 §6이 이미 이관 후 ④Boss Core Gate Decision 순서를 채택 ② 이관팩 세 숫자 보존 ③ 파쇄자=어휘 무확장이라 A의 첫 검증이 저위험 ④ A를 팩 호환 명명으로 지으면 B 승격 경로가 남는다. 권장 시퀀스: **Decision(이관 후) → p14=A안 코어 개정+재-baseline 의식 → Iron Crusher Combat 01 → (어휘 +1~2 포함) Thirst Abyss Combat 01.**
