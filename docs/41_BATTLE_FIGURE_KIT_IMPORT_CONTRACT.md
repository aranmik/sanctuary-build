## A. 나라님 의도 정본 (✅ 잠금)

- **성소판은 이미 재미있다.** 3보스의 재미(판단/화력/인내), 사제 스킬 구성, 동료/파티 구성 확장은 충분히 가능성이 있다.
- **이번 개선의 목적은 재미 보강이 아니라 보는 맛과 첫인상이다.**
- **처음 보는 사람에게도 전투가 귀엽고 예쁘고 게임답게 보여야 한다.**
- **피규어와 행동선은 단순 장식이 아니라 전투 문법을 전달해야 한다.**

> **성소판은 이미 재미있다. 이제 필요한 것은 그 재미를 처음 보는 사람에게 전달하는 보는 맛이다.**
> **미니피규어는 귀엽고 예뻐야 하지만, 동시에 전투 문법이 읽혀야 한다.**

현재 실측: 성소판 스테이지에는 이미 소형 피규어 4종(`fig-war/rog/mage/sham` — 몸통 그라디언트+오라+링+플래시, 42×56px)과 보스 이모지 아바타(☠️)가 있다. 이들은 "위치 표시"까지는 해내지만, "귀엽고 예쁘고 게임답게"의 기준에는 못 미친다. 본 계약은 이들을 대체할 **성소판 전용 정장**의 규격이다.

## B. 이식이 아니라 성소판식 재구성 (✅ 잠금)

**원칙**:
- 옆 프로젝트 코드를 **통째로 복사하지 않는다**.
- 옆 프로젝트의 **완성된 CSS 구조와 규칙을 참고**한다.
- 성소판 전용 **클래스, DOM, 파츠, 네임스페이스를 새로 만든다**(§E).
- **성소판 전투 화면의 비율과 UI 구조를 존중**한다 — 성소판 스테이지는 세로 390 기준 높이 ±104px(소형 뷰 62px)의 얇은 띠다. 옆 프로젝트의 큰 전장 좌표계를 가져오면 성소판 UI(HUD/경고 바/카드/스킬 패널)가 밀려난다.
- 성소판의 **보스/동료/사제 문법에 맞는** 미니피규어 정장을 만든다 — 사제는 공격하지 않고, 보스는 세 가지 질문이다.

> **가져올 것은 코드 덩어리가 아니라 조립형 미니피규어의 규칙이다.**

## C. 옆 프로젝트에서 가져올 원칙 (감사 보고서 기반 · ✅ 잠금)

1. **3층 transform 분리** — 한 유닛의 변형을 세 층으로 나눠 서로 간섭하지 않게 한다:
   - **배치/원근 층**: 전장 위 위치·스케일(뒤는 작게/앞은 크게).
   - **피격·행동 반응 층**: 맞았을 때 흔들림, 행동할 때 기울어짐 — 배치를 건드리지 않고 반응만.
   - **피규어 idle/호흡 층**: 가만히 있어도 살아 있는 미세 호흡 — 반응과 독립.
   - 성소판 매핑: §F의 `.sb-unit`(배치) / `.sb-react`(반응) / `.sb-fig`(idle) 3층.
2. **조립형 part 구조** — 받침 / 몸통 / 머리 / 손 오브젝트 / 역할 장식 / 상태 오버레이 / 행동선 앵커. [39 §E] 부품 7계층과 정합 — 이번엔 그것을 DOM 규격으로 확정한다.
3. **보스/몬스터 6파츠 구조 참고** — shadow / extra / role / body / head / ears. 색·장식 교체만으로 개체가 늘어나는 구조. 성소판은 ears 대신 **aura**(보스 오라)를 여섯 번째 슬롯으로 둔다(성소판 보스는 짐승이 아니라 재앙이다).
4. **색 변수 기반 개체 정체성** — 유닛마다 4색 변수: **main**(주색) / **edge**(윤곽·명부) / **deep**(암부) / **glow**(발광·오라). 새 보스/변주 보스 = 같은 파츠 + 다른 4색 + 다른 role 장식([39 §F] "같은 몸체+다른 장식=변주"와 정합).
5. **FX/행동선은 피규어 앵커 기반으로 재작성** — 옆 프로젝트 fx 레이어를 복붙하지 않고, 성소판식 인터페이스(피규어의 `.sb-anchor`에서 출발/도착)로 새로 쓴다. 문법 어휘(§I): 공격 / 주시 / 도발 / 차단 준비 / 보호막 / 회복·정화·수습 임펄스.

## E. 성소판 전용 네임스페이스 (✅ 제안 확정)

**prefix: `sb-`** (Sanctuary Build)

**왜 네임스페이스가 필요한가**:
- 기존 프로젝트의 `.body`, `.head`, `.part`, `.monster` 같은 **일반명 클래스는 충돌 위험이 높다**.
- 성소판은 **단일 index.html** 구조 — 클래스 충돌이 곧 UI 파손으로 직결된다(성당/길드/전투가 같은 문서 안에 산다).
- 따라서 성소판 전용 prefix로 전투 피규어 계층 전체를 격리한다.
- ※ dev 하네스의 sandbox 변수명 `sb`(JS)와는 무관 — 한쪽은 CSS 클래스, 한쪽은 검증 스크립트 내부 변수로 충돌 없음.

**권장 클래스 세트**:

| 클래스 | 역할 |
|---|---|
| `.sb-unit` | 유닛 루트 — 배치/원근 층 |
| `.sb-react` | 피격·행동 반응 층 |
| `.sb-fit` | 스케일/정렬 보정 층 |
| `.sb-fig` | 피규어 본체 — idle/호흡 층 |
| `.sb-part` | 파츠 공통 |
| `.sb-shadow` | 받침 그림자 |
| `.sb-body` | 몸통 |
| `.sb-head` | 머리 |
| `.sb-hand` | 손 오브젝트(방패/단검/지팡이/구체) |
| `.sb-role` | 역할 장식(문양/토템/망치) |
| `.sb-extra` | 추가 장식(저주 실/리벳/기포) |
| `.sb-aura` | 오라(발광·상태 배경) |
| `.sb-anchor` | 행동선 출발/도착 앵커 |
| `.sb-line` | 행동선(성소판식 fx 인터페이스) |
| `.sb-impulse` | 짧은 임펄스(공격/수습 결과) |

```html
  <div class="sb-react">
    <div class="sb-fit">
      <div class="sb-fig">
        <span class="sb-part sb-shadow"></span>

        <span class="sb-part sb-hand"></span>
        <span class="sb-part sb-role"></span>

  </div>

```

(예시이며 본 카드에서 구현하지 않는다.)

**골격 원칙** (✅ 잠금):
- **DOM은 읽기 쉬워야 한다** — 계층 이름만 보고 무슨 층인지 알 수 있게.
- **파츠 순서는 z-index와 같다** — 마크업 순서 = 쌓임 순서(shadow 맨 아래 → aura 맨 위).
- **파츠 수는 모바일 가독성을 위해 제한한다** — 유닛당 파츠 7개 이내.
- **얼굴보다 실루엣이 먼저다.**
- **캐릭터별 핵심 읽기 포인트는 2~3개까지만 둔다** ([39 §F]와 동일 원칙).

## G. 동료 피규어 방향 (현재 런타임 4동료 기준 · ✅ 잠금)

| 동료 | 전투 문법 | 피규어 방향 | 읽기 포인트(≤3) |
|---|---|---|---|
| **전사** | 탱커 · 버티기 · 강타를 받는 대상 · 보호막의 주요 대상 | 넓은 몸통 · 방패 · 낮은 중심 · 단단한 받침 · 강타선/보호선 앵커 | 방패 · 넓은 실루엣 · 전열 받침 |
| **도적** | 차단 · 빠른 개입 · 낙인 정화 후 차단 성공 | 얇은 몸통 · 단검 · 기울어진 자세 · 차단 준비 앵커 | 단검 · 빠른 실루엣 · 차단선 앵커 |
| **주술사** | 지원 · 자원/유지 · 장기전 보조 | 지팡이 또는 토템 · 작은 오라 · 유지선 앵커 | 토템/지팡이 · 원형 오라 · 지원자 느낌 |
| **마법사** | 원거리 딜러 · 화력 · 보스 압박을 끝내는 딜 축 | 지팡이/구체 · 투사체 방향성 · 충전/발사 임펄스 | 구체 · 캐스팅 오라 · 원거리 축 |

- **향후 7동료**는 이름만 문서 후보로 열어둔다([37 §G] 7명 확정 방향 준수) — 성기사/사냥꾼/수도사 등 **후보명 미확정·구현 ⛔**. 4동료 정장이 검증된 뒤, 같은 골격(§F)+다른 손 오브젝트/4색으로 확장한다.
- 사제(나)는 카드 패널의 주체이지 전장 피규어가 아니다 — 전장에 세울지는 별도 결정(현재 무대에 사제 피규어 없음 · 현행 유지).

## H. 보스 피규어 방향 (3보스 = 3실루엣 · ✅ 잠금 — [39 §D] 계승)

### H-1. 모르가스 — 꼬는 자
- **전투 문법**: 낙인 · 정화 · 차단 연결 · **연결을 꼬는 보스**.
- **피규어 방향**: 가늘고 비틀린 실루엣 · 낙인 문양 · 실/끈/저주선 · 직접 때리는 자가 아니라 연결을 꼬는 자.
- **핵심 파츠**: `sb-body` 가는 몸 / `sb-head` 가면·눈 / `sb-role` 낙인 문양 / `sb-extra` 떠 있는 저주 실 / `sb-aura` 어두운 꼬임 오라.

### H-2. 강철의 파쇄자 — 부수는 자
- **전투 문법**: 강타 · 탱커 체크 · 보호막 · 전선 압박.
- **피규어 방향**: 큰 사각형 몸통 · 금속 테두리 · 망치/파쇄 무기 · 무거운 받침 · **방패/전사 쪽으로 압력을 만드는 보스**.
- **핵심 파츠**: `sb-body` 큰 금속 블록 / `sb-head` 철가면·투구 / `sb-hand`(또는 `sb-role`) 망치 / `sb-extra` 리벳·어깨 장갑 / `sb-shadow` 무거운 압력 그림자 / `sb-aura` 둔중한 압박 오라.

### H-3. 갈증의 심연 — 마시는 자
- **전투 문법**: 마나 압박 · 소모 · 누수 · 버티기.
- **피규어 방향**: 낮고 퍼진 액체형 · 심연 핵 · 안개/기포 · **파티 전체를 천천히 빨아들이는 느낌**.
- **핵심 파츠**: `sb-body` 낮은 액체 덩어리 / `sb-head` 심연 핵·눈 / `sb-extra` 안개·기포 / `sb-role` 흡수구·마나 누수 / `sb-shadow` 번지는 웅덩이 / `sb-aura` 마른·흡수 오라.

세 보스의 4색 변수 예시 방향: 모르가스(main 핏빛/glow 보라) · 파쇄자(main 강철/glow 주황 불꽃) · 심연(main 심청/glow 청록) — 확정은 구현 카드에서.

## I. 행동선/임펄스 문법 (✅ 잠금 — 피규어가 먼저)

**원칙**:
- 행동선은 **버튼을 가리키지 않는다** ([39 §B] 영구 원칙).
- 행동선은 **전장 객체 사이에서 발생**한다.
- 행동선은 **전투 문법을 알려준다**.
- 행동선은 **피규어의 anchor에서 출발하거나 도착**해야 한다.
- **선만 보이면 실패다. 피규어와 함께 읽혀야 한다.**

**최소 행동 문법 6종**:

1. **공격 중** — 무기/손 오브젝트(`sb-hand`)에서 대상 쪽으로 짧은 임펄스.
2. **주시 중** — 보스/대상 사이의 예고선 + 대상 위협 상태 표시.
3. **도발 중** — 탱커 중심의 끌어당김/고정 오라.
4. **차단 준비** — 도적에서 보스 시전선으로 이어질 준비선. 낙인/정화 상태에 따라 흐림/복구([39 §G-2] 연결선 문법).
5. **보호막** — 대상 몸이 아니라 **보호층/방패층이 압박을 받는** 표현([39 §G-4] 보호선=결과).
6. **회복/정화/수습 임펄스** — 사제에서 대상에게, "정답 지시"가 아니라 **수습 결과**로 보이는 짧은 빛([39 §G-5]).

[39 §G]의 5종 선 분류(압박/연결/누수/보호/수습)는 유지된다 — 본 §I는 그 선들이 **피규어 앵커 위에서** 어떻게 발생하는지의 문법이다.

## L. 현재 HOLD 상태 정리 (⏸️)

- **`Iron Crusher Action Line Proto 01`은 기능 검증은 되었지만 커밋 보류/HOLD.**
- 검증됐던 것(참고 기록): 파쇄자 전용 강타 압박선 4상태(예고/임박/보호막 걸림/잔상) · 기존 시전선 핸드오프 · 텍스트 예고 무변 · CORE 무접촉 · 스모크 3종 바이트 동일 · 모르가스/심연 음성 · 390px overflow 0.
- **HOLD 이유**:
  - 선의 의도는 맞지만, **나라님이 원하는 첫인상/보는 맛 기준에는 아직 부족함**.
  - **피규어 조형 없이 선을 얹는 방식은 성소판의 최종 전투 비주얼 방향과 다름** — 이모지 해골과 원형 몸통 사이의 선은 문법은 전달해도 "게임답게 예쁘다"에 못 미친다.
  - **먼저 성소판 전용 미니피규어 정장/규격을 세워야 함**(본 계약).

> **현재 HOLD의 이유는 기능 실패가 아니라, 보는 맛의 기준을 더 높이기 위해서다.**
