> **핵심 문장 3줄**
> 1. **마을은 전투 전 준비 공간이다.** 전투에서 무엇을 수습할 수 있는지는, 마을에서 무엇을 이해하고 챙겼는지로 결정된다.
> 2. 마을은 정답을 주지 않는다 — **내가 어떤 수습 가능성을 챙겼는지 확인**하는 곳이다.
> 3. 이 문서는 구현이 아니라 **계약**이다. 성당/여관/발자취/동료 구조가 각자의 역할로 분리되도록 기준을 세운다(구현 0).

## A. 마을의 핵심 역할 (잠금)

- **마을은 전투의 정답을 주는 곳이 아니다.**
- 마을은 **내가 어떤 수습 가능성을 챙겼는지 확인하는 곳**이다.
- **성당 / 여관 / 토벌 게시판 / 발자취는 역할이 분리되어야 한다.** (각 시설은 하나의 목적만)
- **시설 목적을 흐리는 흐름을 피한다** — "성당에서 전투 시작", "여관에서 전투 시작" 같은 진입은 금지 방향.
- **출격은 토벌 게시판(보스 선택 화면)에서만 이루어진다.**

각 시설의 한 문장 역할:
| 시설 | 한 문장 역할 |
|---|---|
| 성당 | 이번 전투를 **어떤 6개 수습 도구**로 해결할지 정한다 |
| 여관 | 동료의 **역할과 실수**를 이해하고 관리한다 |
| 토벌 게시판 | 보스를 고르고 **출격**한다 (질문을 만난다) |
| 발자취 | 지나온 **원정의 기록**을 되짚는다 |

## B. 성당 UX 문제 정의 (현행 실측)

**현재 구조**(renderChapel · index.html ~1236): 6칸 장착 슬롯(`.ldSlots`) → 9스킬 목록(세로 스크롤) → `▶ 이 구성으로 전투 시작`(chapel-start) + `🏛️ 저장하고 마을로`(chapel-save).

- **스킬 목록을 아래로 스크롤하면 상단 장착 6칸이 화면 밖으로 사라진다.** 지금 무엇을 꼈는지 보면서 아래 후보를 비교할 수 없다.
- 현재 9스킬에서는 큰 문제가 아니지만, **향후 20개 전후가 되면 비교/장착 판단이 어려워진다.**
- 플레이어가 **"무엇을 장착했고 무엇이 빠졌는지" 계속 볼 수 있어야 한다.**
- 성당은 **스킬 창고가 아니라 "이번 전투를 어떤 6개 수습 도구로 해결할지 정하는 곳"**이다.

## C. 성당 목표 레이아웃 (설계)

1. **상단 고정 영역** — 현재 장착 스킬 6칸. **스크롤해도 항상 보임.** 각 칸 상태 3종 구분: 비어 있음 / 장착됨 / 선택 중.
2. **구성 요약 영역** — 현재 loadout의 강점/빈틈을 **1~3줄로 요약**(§E "성당 메시지" 정책). 예: 정화 없음 · 보호 부족 · 지속 회복 부족 · 광역 회복 있음.
3. **카테고리 탭** — 스킬 수가 20개 전후로 늘어도 한 화면에서 찾도록 분류(§D). 후보 4탭: 치유 · 보호 · 정화·유틸 · 지속·자원.
4. **스킬 카드 목록** — 선택한 카테고리의 스킬만 표시. 카드에 **핵심 역할을 짧게** 표기("어떤 전장 문제를 수습하는가").
5. **하단 버튼** — `저장하고 마을로`(유지). **`이 구성으로 전투 시작`은 제거 후보** — 전투 시작은 토벌 게시판/보스 선택 흐름에 맡김(§A 시설 목적 분리).

- 실측 각주: 상단 6칸(`.ldSlots`)과 상태 클래스(`.ldSlot.filled/.empty`)는 이미 존재 — 상단 고정(sticky)과 "선택 중" 상태만 추가하면 되는 지형. `chapel-start` 제거는 `enterBattle(1)` 경로(성당 임시 시작)를 끊는 것이며, 정식 출격 경로(sortie→enterBattle(SELECTED_BOSS))는 무관.

## D. 스킬 카테고리 기준 (전장 문제 기준 · 판타지 속성 아님)

> 카테고리는 "무엇 속성인가"가 아니라 **"어떤 전장 문제를 수습하는가"**로 나눈다.

### 치유 — 직접 HP를 복구
빠른 치유 · 강력 치유(깊은 치유) · 치유의 기도(공명의 기도) · 빛의 고리(후보)

### 보호 — 피해가 들어오기 전에 받침
보호막 · 수호 계열(후보)

### 정화·유틸 — 전장 사고 / 상태 이상 / 행동 기회 문제를 풂
정화 · 대정화(후보) · 낙인/차단 연계 대응

### 지속·자원 — 장기전 / 마나 / 유지력 / 버티기
소생 · 기도씨앗 · 성심 집중

- 현재 9스킬 매핑(참고): 치유=빠른 치유·강력 치유·치유의 기도 · 보호=보호막 · 정화·유틸=정화 · 지속·자원=소생·기도씨앗·성심 집중 · (긴급 구원=치유/보호 경계 — 착공 카드에서 확정). 20스킬 확장 시 각 탭이 5개 안팎으로 균형.

## E. loadout 메시지 정책 (3단 분리 · 잠금)

메시지를 세 화면에 나눠, 성당이 시끄러워지지 않게 한다:

| 화면 | 무엇을 보여주나 | 예 |
|---|---|---|
| **성당** | 내 구성의 **일반 성향**(보스 무관) | "정화 없음 — 상태 사고 대응이 약합니다." · "보호 부족 — 큰 단일 피해를 미리 받치기 어렵습니다." · "지속 회복 부족 — 장기전에서 마나 부담이 커질 수 있습니다." · "광역 회복 있음 — 파티 전체 흔들림을 수습할 수 있습니다." · "큰 치유 과다 — 순간 복구는 강하지만 장기전 부담이 커질 수 있습니다." |
| **출격**(전투 전 거울) | 이 **보스와 만났을 때의 위험**(보스별 구체) | 모르가스: 정화 없으면 낙인/차단 연계 흔들림 · 파쇄자: 보호막 없으면 강타 전선 흔들림 · 심연: 지속/자원 도구 없으면 마나 먼저 마름 (→ [35] 이미 구현) |
| **결과**(전투 후 거울) | **실제로 흔들린 지점** | [34] 보스별 결과 거울 (→ 이미 구현) |

- 원칙: **성당=내 구성의 일반 성향 / 출격=이 보스와의 위험 / 결과=실제 흔들린 지점.** 성당 메시지는 보스별 정답이 아니라 성향 진단이다.
- 실측 정합: 출격(35)·결과(34)는 이미 이 3단 구조의 뒤 두 칸을 채웠다. 본 계약은 **앞 칸(성당 일반 성향)**을 정의한다 — 착공은 Chapel Loadout UI 01.

## F. 여관 UX 방향

- **여관은 동료/직업 역할 정보를 보여주는 주 창구**가 된다 — "주술사가 마나 회복을 돕는다 / 도적이 차단한다 / 전사가 전선을 버틴다"를 플레이어가 배우는 곳.
- 다만 **실제 여관 UI 개편은 7직업/성장/특성 방향이 더 잠긴 후 구현**한다.
- **지금 단계에서는 여관의 역할을 문서로 정의하고, 구현은 보류**한다.
- 여관은 장기적으로 **동료 선택 · 역할 설명 · 특성 확인 · 성장 관리**와 연결될 수 있다.
- 연결: 여관의 역할 정보는 [33] 실패 3층 중 **1층(파티 구성)**의 학습 창구다 — 왜 이 동료가 필요한지 여관에서 이해하고, 출격 경고/결과 거울에서 확인한다.

## G. 발자취 / 모험의 기록

**현행 실측**: 마을 상단 `#village-status`(index.html ~395/1218)에 "원정 클리어 N회 · 최고 기록 X초"가 상시 노출.

- **마을 상단은 현재 준비/선택 상태 중심으로 가볍게 유지**한다.
- 누적 기록은 **`발자취`**(유키 추천 이름)라는 별도 메뉴로 분리한다. (후보: 발자취 / 모험의 기록 / 성소 기록 / 원정 기록 → **발자취** 채택.)
- 발자취에 담을 수 있는 기록: **보스별 클리어 · 최고기록 · 원정 횟수 · 최고 생존 시간 · 마나 바닥 후 버틴 시간** 등.
- **이번 카드에서는 구현하지 않는다.** (착공: Footprints Record Panel 01 · SAVE 확장 없이 현재 `SAVE.data()`의 clears/best를 재배치하는 수준부터.)

## H. 동료 구조 방향 (나라님 고민 · 단기/장기 분리)

**고민 두 갈래**:
1. **7개 직업 = 확정 동료** — 7명의 고정 파티원을 성장/특성 관리.
2. **직업별 여러 영웅 모집** — 예: 도적A 아란=차단형 · 도적B 배티=딜형.

**유키 추천 방향**:

### 단기 / 2차 초반 — 7명의 확정 동료를 우선한다
- 성소판은 **사제가 파티를 이해하고 수습하는 게임**이다.
- 동료가 랜덤 부품보다 **"내가 아는 친구들"**일수록 파티 감정과 전장 읽기가 강해진다.
- 직업별 다중 영웅을 너무 빨리 열면 **정답 영웅 수집 게임**으로 기울 수 있다.

### 장기 확장 후보 — 같은 직업의 다른 영웅 모집
- 예: 아란(차단형 도적) · 배티(딜형 도적) · 시오(생존형 도적).

> **성소판의 동료는 처음에는 7명의 확정 파티원이다.**
> **플레이어는 이들의 역할과 실수를 이해하며 사제로 수습한다.**
> **같은 직업의 다른 영웅 모집은 장기 확장 후보로 남긴다.**

- 실측 각주: 현재 데모는 4동료 고정(war/rog/mage/sham · CFG.allies). 7직업(기사/궁수/흑마 추가)은 [22] 로스터에 설계됨 — 본 계약은 "7명 확정 우선"을 **방향으로 잠그되**, 착공은 Companion Identity Decision 01.

## I. 구현 순서 제안 (착공은 유키 발주 · 전부 미구현)

1. **Chapel Loadout UI 01** — 상단 장착 슬롯 고정 · 카테고리 탭 · 20스킬 대응 구조 · `저장하고 마을로` 중심 · 성당 내 전투 시작 버튼 제거 후보.
2. **Battle Visual Identity Contract 01** — 보스/동료 미니 피규어 및 행동선 시각 문법 설계.
3. **Companion Identity Decision 01** — 7명 확정 동료 vs 직업별 다중 영웅 모집 구조 결정.
4. **Inn Companion Role Sheet 01** — 동료 역할 설명 UI (단, 7직업/특성 방향 결정 후 구현).

- 순서 근거: 1(성당)은 이미 손맛이 검증된 로드아웃의 확장이라 저위험 첫 타. 3(동료 정체성)은 4(여관)·H항의 전제라 여관보다 먼저. 2(전투 비주얼)는 독립 병행 가능. 5(발자취)는 SAVE 확장 없이 재배치라 가벼움.
